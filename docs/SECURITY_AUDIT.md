# Security audit — v16

Дата проверки: 2026-06-23.

## Scope

Проверены сетевые запросы, URL, ZIP и файловый ввод, хранение сессий и секретов, запуск локальных процессов, недоверенный LLM-контент и новая парольная защита.

## Password and encryption design

- при отсутствии конфигурации пароля запрос при запуске не появляется;
- пароль не хранится;
- 16-байтовая случайная соль и `scrypt` (`N=32768`, `r=8`, `p=1`) используются для получения ключевого материала;
- проверка пароля выполняется через HMAC и `compare_digest`;
- сессии шифруются AES-256-GCM;
- для каждого сохранения создаётся новый 96-битный nonce;
- AES-GCM одновременно обеспечивает конфиденциальность и проверку целостности;
- ключ находится только в RAM и забывается при штатном закрытии или ручной блокировке;
- старые JSON-сессии мигрируют в `.llms` после установки или успешного ввода пароля;
- смена и удаление пароля используют промежуточный проверенный JSON, чтобы снизить риск потери данных при прерывании операции.

## Earlier fixes retained

- безопасная распаковка ZIP без `extractall` и защита от Zip Slip/ZIP bomb;
- API-ключи не сохраняются в сессиях;
- SSRF-защита загрузчика публичных веб-страниц;
- недоверенные файлы и страницы помечаются для модели как данные, а не инструкции;
- запуск batch-команд без `shell=True`;
- проверка пользовательских LLM URL.

## Remaining risks

- парольный экран нельзя считать защитой от пользователя с полным контролем над ОС или изменённым EXE;
- ключ присутствует в памяти во время работы приложения и Python не гарантирует физическое затирание immutable `bytes`;
- исходные файлы проекта не шифруются;
- имя сессии видно по имени `.llms` файла;
- слабый пароль остаётся уязвимым для офлайн-перебора, несмотря на `scrypt`;
- пользователь может выбрать исполняемый файл llama.cpp;
- модель может вернуть опасный код;
- HTTP без TLS не защищает удалённое соединение;
- сторонние поисковые API получают запросы пользователя.
